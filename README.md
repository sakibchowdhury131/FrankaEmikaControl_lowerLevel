# Panda Polynomial Trajectory Planner + Executor

Two-stage system with a clean Python/C++ split:

```
Python (offline, any machine)      C++ (real-time, robot PC)
──────────────────────────────     ────────────────────────────────────
planner/main_plan.py          →    executor/src/panda_executor.cpp
  • compute coefficients               • load trajectory.csv
  • RNEA torque check (Pinocchio)      • move_to_start (position control)
  • singularity check + via-pts        • 1 kHz torque control loop
  • self-collision check               • runtime monitor + E-stop
  • export trajectory.csv              • libfranka native API
```

## Step 3 — Execute on real robot

```bash
# From executor/build/:
./panda_executor ../../trajectory.csv 192.168.1.1
```

The executor will:
1. Connect to the robot and call `automaticErrorRecovery()` to clear any existing error state
2. Move to `q_start` at 12% speed (position control), then settle for 1.5 s
3. Print trajectory info and wait for ENTER
4. Stream torques at 1 kHz using the control law below
5. E-stop automatically on contact / singularity / velocity limit / robot error

### Control law

```
tau_ff  = M(q_actual)·ddq_des(t) + C(q_actual, dq_actual)·dq_actual + g(q_actual)
tau_cmd = tau_ff + Kp*(q_des(t) - q) + Kd*(dq_des(t) - dq)
```

The feedforward is computed **online at the robot's actual state** using
libfranka's built-in model (`model.gravity`, `model.coriolis`, `model.mass`),
with `ddq_des` from the planned CSV.  This eliminates the phase-mismatch problem
where a time-indexed CSV feedforward switches to a braking torque while the robot
is still accelerating.  The PD terms provide additional correction for residual
model error.

Execution is split into three phases:

| Phase      | Duration    | Behaviour                                                    |
|------------|-------------|--------------------------------------------------------------|
| Ramp-up    | 500 ms      | Hold `q_start`, ramp `tau_ff` from 0 → `traj[0].tau`        |
| Tracking   | trajectory (or N/2 in throw mode) | Follow CSV at 1 kHz    |
| Ramp-down  | 500 ms      | Ramp `tau_ff` from last trajectory torque → gravity-only     |

#### Throw mode

Set `kThrowMode = true` in `panda_executor.cpp` to stop trajectory playback at
the **peak-velocity midpoint** (N/2 samples) rather than running the full
deceleration phase.  Use this for ball-throwing tasks where the ball should be
released at peak arm velocity.  The arm continues moving under inertia after
the trajectory index freezes; `kThrowMode = false` runs the complete arc.

## Important notes

### Torque feedforward and gravity
The RNEA torques stored in the CSV include gravity:
```
tau = M(q)·q̈ + C(q,q̇)·q̇ + g(q)
```
libfranka's `robot.control(torque_callback)` does **NOT** add gravity
internally — you are responsible for the full torque. This is correct as
implemented. If you use a hybrid impedance controller that handles gravity
internally, subtract `pin.computeGeneralizedGravity(model, data, q)` before
writing to the CSV.

### Via-point velocity
The current implementation forces **zero velocity at the via-point**
(two rest-to-rest segments). This is safe and simple but increases total
duration. For smooth pass-through via-points with non-zero intermediate
velocity, the polynomial boundary conditions need to be extended to a
12×12 system matching velocity and acceleration at the junction — a
straightforward extension of the same math.

### Real-time requirements
libfranka requires a PREEMPT_RT patched kernel for reliable 1 kHz control.
Without it you will see `franka::ControlException: communication_constraints_violation`.
See: https://frankaemika.github.io/docs/installation_linux.html

## Ball-throw module (`ball_throw/`)

The `ball_throw/` sub-module extends the base trajectory planner with a
purpose-built pipeline for throwing a ball to a specified position in space.
It solves the full problem: task-space IK for the release state, two-phase
5th-order polynomial planning under hardware limits, RNEA torque verification
with payload, CSV export for the C++ executor, and post-execution verification
of what the robot actually achieved.

### Concept

A ball throw is broken into two kinematic phases connected at a single
**release instant**:

```
Phase 1 (rest → release)          Phase 2 (release → stop)
q_start (v=0, a=0)    ──────→    q_release (dq_release, a=0)    ──────→    q_stop (v=0, a=0)
                   5th-order poly                              5th-order poly
```

At the release instant the arm has exactly the joint velocity `dq_release`
needed to produce the desired end-effector velocity `V_EE_RELEASE`.
Acceleration is forced to zero at the release boundary — the arm is neither
speeding up nor slowing down when the ball leaves, which keeps the throw
velocity clean and predictable.

After release, `kThrowMode = true` in the C++ executor lets the arm follow
Phase 2 smoothly to `q_stop` rather than freezing at peak velocity.

### Step 1 — Configure the throw

Open `ball_throw/main_throw.py` and edit the **USER CONFIGURATION** section:

| Parameter        | Type       | Description                                                  |
|------------------|------------|--------------------------------------------------------------|
| `Q_START`        | `[7] rad`  | Joint config where the robot starts (at rest)               |
| `Q_STOP`         | `[7] rad`  | Joint config the robot returns to after the throw           |
| `P_RELEASE`      | `[3] m`    | End-effector position at the moment of ball release (world) |
| `V_EE_RELEASE`   | `[3] m/s`  | End-effector linear velocity at release — sets throw direction and speed |
| `Q_RELEASE_INIT` | `[7] rad`  | Initial guess for IK (should be close to the release pose)  |
| `PAYLOAD_MASS`   | `float kg` | Mass of ball + gripper attachment                           |
| `PAYLOAD_COM`    | `[3] m`    | Centre of mass of payload in the flange frame               |

**Choosing `V_EE_RELEASE`:** keep the magnitude below ~1.5 m/s for most
configurations. The Panda's joint velocity limits (2.175 rad/s for joints 1–4)
constrain achievable end-effector speed. Too fast causes the velocity IK to
exceed joint limits and planning will fail with a clear error message.

Rule of thumb: start at 0.5 m/s and increase until you hit warnings.

### IK internals

**Position IK** (`ik_solver.py:ik_position`) uses iterative CLIK with a
damped Jacobian pseudo-inverse:

```
dq = Jᵀ (J Jᵀ + λI)⁻¹ · e_pos
q  ← clip(q + α·dq,  q_min, q_max)
```

Only position (3-DOF) is controlled. Orientation is free — correct for
ball-throwing where the release point matters but wrist orientation does not.

**Velocity IK** (`ik_solver.py:ik_velocity`) is a single linear solve
(least-norm solution):

```
dq = J† · v_ee  =  Jᵀ (J Jᵀ + λI)⁻¹ · v_ee
```

The result is checked against joint velocity limits immediately; if any joint
exceeds its limit the planner raises a descriptive error with fix suggestions
before any trajectory is built.

### Polynomial math internals

Each phase is a 5th-order polynomial per joint with **general boundary
conditions** `(q₀, v₀, a₀) → (qf, vf, af)`. The 6 coefficients are solved
in closed form by substituting the `t=0` conditions directly and reducing to
a 3×3 system for the higher-order terms. See `poly5_general.py` for the
derivation.

Minimum duration `T_min` is found by **bisection** — the feasible set is
`[T_min, ∞)` (monotone property: all derivatives scale down as T increases),
so bisection converges in ~14 iterations to 1 ms precision. A 5% safety
margin is applied to the bisection result.

### Torque check with payload

After sampling at 1 kHz, RNEA (`pinocchio.rnea`) is called at every time
step with the payload inertia added to joint 7. If any torque exceeds the
hardware limit (`TAU_MAX`), `T` is scaled up by 20% and the phase is
re-sampled — up to 30 iterations. This guarantees the trajectory is
dynamically feasible with the ball attached before the CSV is written.
