## Dependencies

### Python (planner)

A virtual environment is already provided in `env/`. Activate it before running any planner commands:
```bash
source env/bin/activate
```

Or install into your own environment:
```bash
pip install numpy pytest
conda install pinocchio -c conda-forge   # for dynamics + collision checks
```

### C++ (executor)
- **libfranka в‰Ґ 0.9** вЂ” https://github.com/frankaemika/libfranka
- **Eigen3** вЂ” `sudo apt install libeigen3-dev`
- **CMake в‰Ґ 3.10**, **GCC/Clang with C++17**
- **Real-time Linux kernel** (PREEMPT_RT) вЂ” required by libfranka for 1 kHz control

Install libfranka from source:
```bash
sudo apt install build-essential cmake git libpoco-dev libeigen3-dev
git clone --recursive https://github.com/frankaemika/libfranka
cd libfranka && mkdir build && cd build
cmake -DCMAKE_BUILD_TYPE=Release -DBUILD_TESTS=OFF ..
cmake --build . && sudo make install
```

### URDF (for Pinocchio)
```bash
# With ROS:
rospack find franka_description
# Copy the panda/ subfolder as planner/franka_panda/

# Without ROS:
git clone https://github.com/frankaemika/franka_ros
cp -r franka_ros/franka_description/robots/panda planner/franka_panda
```

### Real-time scheduling capability

libfranka requires real-time scheduling priority. After building, grant the
binary the capability to set `SCHED_FIFO` without running as root:

```bash
sudo setcap cap_sys_nice+ep build/panda_executor
```

Alternatively, add your user to the `realtime` group and raise `rlimits` via
`/etc/security/limits.conf`.

## Step 3 вЂ” Execute on real robot

```bash
# From executor/build/:
./panda_executor ../../trajectory.csv 192.168.1.1
```

The executor will:
1. Connect to the robot and call `automaticErrorRecovery()` to clear any existing error state
2. Move to `q_start` at 12% speed (position control), then settle for 1.5 s
3. Print trajectory info and wait for ENTER
4. Stream torques at 1 kHz using the control law below
5. E-stop automatically on contact / singularity / velocity limit / robot error

### Control law

```
tau_cmd = tau_ff(t) + Kp*(q_des(t) - q) + Kd*(dq_des(t) - dq)
```

where `tau_ff` comes from the CSV. The PD terms correct for model inaccuracies
during tracking. Execution is split into three phases:

| Phase      | Duration    | Behaviour                                      |
|------------|-------------|------------------------------------------------|
| Ramp-up    | 500 ms      | Hold `q_start`, ramp `tau_ff` from 0 в†’ `traj[0].tau` |
| Tracking   | trajectory  | Follow CSV at 1 kHz                            |
| Ramp-down  | 150 ms      | Ramp `tau_ff` from `traj.back().tau` в†’ gravity |

Edit in `executor/src/panda_executor.cpp`:

|---------------------|-----------|--------------------------------------|
| `kTauExtMax`        | 10.0 Nm   | External torque E-stop threshold     |

| `kSigmaMinWarn`     | 0.04 m/rad| Singularity warning threshold        |
| `kMoveToStartSpd`   | 0.12      | Speed factor for move-to-start       |
| `kRampMs`           | 500 ms    | Torque ramp-up duration              |
| `kRampDownMs`       | 150 ms    | Torque ramp-down duration            |
| `kKp`               | [50,50,50,50,20,20,20] Nm/rad | PD position gain    |
| `kKd`               | [7,7,7,7,3,3,3] NmВ·s/rad     | PD velocity gain    |

## Important notes

### Torque feedforward and gravity
The RNEA torques stored in the CSV include gravity:
```
tau = M(q)В·qМ€ + C(q,qМ‡)В·qМ‡ + g(q)
```
libfranka's `robot.control(torque_callback)` does **NOT** add gravity
internally вЂ” you are responsible for the full torque. This is correct as
implemented. If you use a hybrid impedance controller that handles gravity
internally, subtract `pin.computeGeneralizedGravity(model, data, q)` before
writing to the CSV.

### Via-point velocity
The current implementation forces **zero velocity at the via-point**
(two rest-to-rest segments). This is safe and simple but increases total
duration. For smooth pass-through via-points with non-zero intermediate
velocity, the polynomial boundary conditions need to be extended to a
12Г—12 system matching velocity and acceleration at the junction вЂ” a
straightforward extension of the same math.

### Real-time requirements
libfranka requires a PREEMPT_RT patched kernel for reliable 1 kHz control.
Without it you will see `franka::ControlException: communication_constraints_violation`.
See: https://frankaemika.github.io/docs/installation_linux.html
